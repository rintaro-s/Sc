# **0\. なぜ私はこれを作るのか**

**_VRChatで話していた友達が、ネットワークの揺れのたびにテレポートする。その瞬間、「一緒にいる」という感覚が消える。技術的に解決できるはずなのに、誰もOSレイヤーから直していない。それが許せなかった。_**

VRChatでの会話中、0.5秒のパケットロスがあるたびに相手のアバターが空間的に瞬間移動する。"together" の感覚が、物理法則を無視した座標の上書きによって破壊される。

調べると、原因は単純だった。受信した最新パケットの座標をそのまま適用しているからだ。速度・加速度・IMUデータをパケットに埋め込み、欠落した間を慣性で予測すれば解決できる。なのに誰もアプリケーション層より下で実装していない。

では自分が書く。それが動機のすべてだ。

この問題を突き詰めていくと、構造的な欠陥に行き着いた。VRChatもSNSも、それぞれ「閉じた宇宙」として設計されており、「現実の座標」という共通言語を持っていない。ARグラスで現実を見ながらVRChatの友達と話し、Instagramに投稿された今いる場所の写真も同時に視界に浮かぶ--そんな体験を実現するOSレベルの基盤が、2024年に存在していない。

CR-Bridgeはその基盤を、OS直下から、オープンに、誰でも使えるミドルウェアとして作る。

# **1\. 何が新奇か--「誰もやっていない理由」から考える**

技術要素の個々はすでに存在する。カルマンフィルタも、FlatBuffersも、QUICも、S2 Geometryも。ならばなぜ誰もこの形で組み合わせていないのか。

## **1.1 「SNSに座標を後付けする」という逆転の発想**

既存のARプラットフォーム（Niantic、ARCore等）は「現実にデジタルを貼る」という設計思想で作られている。現実が主、デジタルが従。

CR-Bridgeは逆だ。Twitterのツイートを「空間に浮かぶパケット」として先に定義し、現実の座標系をそこに後からマッピングする。「仮想が主、現実が従」という思想で設計する。これにより、VRChatの「東京駅ワールド」にいる人と、現実の東京駅にいる人と、Instagramの「#東京駅」タグの写真が、一つの空間インデックスに共存できる。

**_SNSの投稿は、ずっと「時系列」で流れてきた。CR-Bridgeはそれを「空間」に変換する最初のミドルウェアだ。_**

## **1.2 「テレポート」を工学的に禁止する**

ゲームエンジンやVRフレームワークは、パケットロス時の挙動を「アプリの問題」として扱ってきた。ネットワーク層は「届けるだけ」、アプリ層は「受け取るだけ」。その分業がテレポートを生む。

ATPプロトコルはこの前提を壊す。パケット自体に速度・加速度・IMU角速度を含め、受信端が「届かなかった間の挙動」を物理的に再現する。欠落は「補間すべき物理状態」として扱われ、アプリ側はテレポートが「アーキテクチャ的に起きない」世界を受け取る。

## **1.3 「セレンディピティの空間」という副産物**

設計の副作用として、意図せず面白い現象が起きる可能性がある。現実の「東京駅の喫茶店」という座標に、VRChatで同じ場所の仮想ワールドにいる見知らぬ人のアバターが「浮遊」し、そこに「#東京駅でラテアートを注文した」というツイートが空間的に重なる。

これは出会いの設計ではない。空間インデックスが生み出す、計算されていないセレンディピティだ。システムが意図せず作り出す「偶然の接触」こそが、新しい社会的価値になりうると考えている。

# **3\. 技術設計--低レイヤーから一気通貫で書く理由**

「Unityで実装すればいい」という問いへの答えから始める。

**_ゲームエンジンの上に乗ったまま「低遅延」を目指すのは、高速道路の渋滞をバスの中で解決しようとするようなものだ。渋滞自体を作り直すには、道路を設計し直すしかない。_**

CR-BridgeはOSのシステムサービスレベル（Linuxデーモン）で動作し、各アプリケーションの入出力を横断する。アプリ層のオーバーヘッドを経由しないため、複数のVRアプリ・SNSブラウザ・AR描画エンジンがゼロコピーで同一の空間データを共有できる。

## **3.1 4層アーキテクチャ**

| **層** | **コンポーネント** | **「なぜこの層か」の理由** |
| --- | --- | --- |
| プロトコル層 | Anti-Teleport Protocol (ATP) | パケットロス対策はアプリではなく伝送プロトコルが解決すべき問題 |
| 計算層 | Spatial Math Accelerator (SMA) | Unity等の汎用エンジンでは100体のSIMD並列変換は保証されない |
| データ構造層 | Shared Memory Spatial Ledger (SMSL) | プロセス間コピーをゼロにしないと累積レイテンシが積み上がる |
| ブリッジ層 | Protocol Bridges | 既存サービスは「空間」を知らない。知らないなら外側から貼り付ける |

## **3.2 Anti-Teleport Protocol (ATP)**

### **問いの立て方**

「パケットが届かなかった間、エンティティはどこにいるべきか」--この問いをプロトコルレベルで定義した設計はほぼ存在しない。ATPはこれを数学的に強制する。

- パケット構造に速度・加速度・角速度（IMUデータ）を含める（FlatBuffers + QUIC Multi-Stream）
- 受信端で拡張カルマンフィルタ（EKF）が欠落区間の状態を毎フレーム推定し続ける
- 次パケット到着時、推定値と実測値をエルミート補間（C1連続）で合流させる。目がわかる「テレポート」は起きない
- FMA命令（Fused Multiply-Add）でカルマンゲイン行列演算を実装、1エンティティあたり < 5μs を実測で達成

## **3.3 Spatial Math Accelerator (SMA)**

### **なぜSIMDで書くか**

VRChatの100体のアバターを、ARグラスの現在の視点から見た座標に変換する処理が毎フレーム必要になる。この計算をUnityのC#に任せると、GCポーズ・スレッドスケジューリング・JITのオーバーヘッドが積み上がる。SMAはこの計算をRustのAVX-512 intrinsicsで直接記述し、L1/L2キャッシュ内で完結させる。

- SoA（Structure of Arrays）レイアウトでエンティティを格納し、AVX-512の16並列float演算を最大活用
- ARグラスのSoC（Snapdragon 8 Gen等）ではNEON intrinsicsで同等実装
- NPU（Neural Processing Unit）に軽量GRUモデルを配置し、50ms先の視線方向を予測、視錐台外データのプリフェッチを最小化する

## **3.4 Shared Memory Spatial Ledger (SMSL)**

ネットワーク受信バッファ → AR描画エンジンの経路でメモリコピーを一切発生させない設計。

- FlatBuffersバイナリをmmapした共有メモリにそのまま配置。パースコストゼロ
- SeqLockで単一ライター・複数リーダーを実現。mutexなしで描画コアが座標を読む
- S2 Geometryセルで地球全体を階層インデックス化。ARグラスの視錐台クエリをO(log N)で処理

## **3.5 Protocol Bridges**

| **接続先** | **プロトコル** | **CR-Bridgeでの変換** |
| --- | --- | --- |
| VRChat | OSC / UDP（公式サポート） | ボーン座標 → FlatBuffersエンティティ → SMSL登録 |
| Twitter/X | Streaming API（Filtered Stream） | ツイート + 位置情報 → ジオコード → S2インデックス登録 |
| Instagram | Graph API（Media with Location） | 位置付きメディア → S2インデックス登録（鮮度管理付き） |
| ARグラス | QUIC → OpenXR Overlay | SMSLから視錐台クエリ → AR描画レイヤーへプッシュ |
| スマートフォンVPS | 特徴点記述子 → サーバー照合 | カメラ映像から世界座標を確定、SMSLに反映 |

# **4\. 5年後の世界と、その前の失敗**

## **4.1 目指す世界**

ARグラスが普及した5〜10年後、人々は「インターネット」と「現実」を別々のウィンドウで見ることをやめる。視界の中に、VRChatの友達のアバターが隣に立ち、Instagramの投稿が建物の壁に浮かび、Twitterの会話が歩道に流れる。

その世界のインフラとなるミドルウェアを、今、OS直下から設計する。NianticはVPSで同じ夢を見ているが、クローズドエコシステムで囲い込む。CR-Bridgeは、誰でも使えるオープンなインフラとして逆張りする。

## **4.2 想定する失敗と、それでもやる理由**

- SNS APIの制限強化：Twitter/XはAPIを段階的に制限している。SNS Bridgeが完全動作しない可能性がある。→ VRChatとARグラスの接続だけでも独立した価値があると判断している
- ARグラスの普及遅延：視界に情報を重ねる体験が市場に受け入れられない可能性がある。→ ミドルウェアとしての技術的価値（テレポート補間・ゼロコピー空間DB）はXRの普及速度に依存しない
- 商業化の不透明さ：このミドルウェアが直接収益を生むモデルは現時点で見えていない。→ OSS基盤として公開し、上位レイヤーのサービス開発者が利用するインフラを目指す

それでも作る。VRChatの友達がテレポートするのが許せないという、その感覚が正しいと信じているから。技術的に解決できるのに誰もやっていないことを放置できない性格だから。

# **7\. 最後に--審査員の方々へ**

この提案の技術的コンポーネントはすべて実在する。カルマンフィルタもSIMDもQUICも、教科書に載っている。「新しい技術」を発明する提案ではない。

**_「誰もこの組み合わせで、このレイヤーで、このユースケースのために書いていない」--それだけが、私がこれを作る唯一の理由だ。_**

VRChatの友達がテレポートするのが許せない。SNSの投稿が時系列にしか存在できないのが惜しい。ARグラスで現実を見ながら、仮想の友人と同じ空間を共有できないのが悔しい。

この感情は、5年後も10年後も変わらないと思っている。だからこのミドルウェアを書く。未踏はその最初の12ヶ月を、最大密度で使う機会だと考えている。
