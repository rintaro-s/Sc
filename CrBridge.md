

**未踏IT人材発掘・育成事業　提案書**

# **1\. 提案概要**

本提案「CR-Bridge」は、VRChat・Twitter/X・Instagram等の既存インターネットサービスと、ARグラス・映像/音声出力等の現実インタフェースをリアルタイムに接続する低レイヤーミドルウェアの開発である。

既存の「メタバース連携」ソフトウェアの多くはアプリケーション層（Unity等ゲームエンジン）で実装されており、コンテキストスイッチ・メモリコピー・ネットワークジッタの3つのオーバーヘッドが累積し、XRの没入感を損なっている。本システムはOS直下のシステムソフトウェア層で動作し、これら3つのオーバーヘッドをアーキテクチャレベルで排除する。

| **項目** | **内容** |
| --- | --- |
| 開発言語 | Rust（メインランタイム）/ C intrinsics（SIMDカーネル）/ WGSL（GPUシェーダー） |
| ターゲットOS | Linux（主）/ Android / Windows（OpenXR経由） |
| 主要技術 | QUIC/HTTP3・FlatBuffers・eBPF・SIMD/FMA・mmap・Kalman Filter・S2 Geometry |
| 提案期間 | 12ヶ月（3フェーズ） |
| ハードウェア要件 | ARグラス（\$200〜500クラス）・一般的なLinux/Androidスマートフォン |

# **2\. 背景と課題**

## **2.1 XR体験を阻む3つの技術的ボトルネック**

現状のクロスリアリティ体験において、以下3つの問題が連鎖的に体験品質を低下させている。

- テレポート問題：パケットロス時、受信した最新座標をそのまま上書きするため、アバターや仮想オブジェクトが空間的に瞬間移動する現象が発生する。VRChatやWebXRで日常的に観察され、没入感を根本から破壊する。
- レイヤー間コピーコスト：ネットワーク受信バッファ→パーサー→アプリケーションモデル→描画エンジンと複数回のメモリコピーが発生する。8K点群・60fpsの空間データでは累積コピーコストが数十msに達する。
- SNSの「無空間性」：Twitter/InstagramはGPS座標を持つコンテンツでも、それを空間的に検索・配信するAPIを提供していない。既存の連携ツールはポーリングで取得したデータをアプリ層で手動マッピングしており、リアルタイム性と拡張性に欠ける。

## **2.2 既存プロダクトとのギャップ**

類似する取り組みとして、NianticのLightship ARDK、GoogleのARCore Geospatial API、MetaのHorizon OSが存在する。しかしいずれも以下の点で本提案とは異なる。

| **プロダクト** | **強み** | **本提案との差異** |
| --- | --- | --- |
| Niantic Lightship | VPS・地球規模の3Dマップ | 閉じたエコシステム。SNS統合・低レイヤーAPIは非公開 |
| ARCore Geospatial | GPS+Street View VPS | Android/Google依存。汎用ミドルウェアとして利用不可 |
| OpenXR Runtime | XRデバイス統一API | 座標同期・SNS統合・予測補間の機能は対象外 |
| VRChat OSC | アバター状態の外部公開 | 単方向・低頻度。AR側へのリアルタイム転送設計がない |
| CR-Bridge（本提案） | 上記全ての橋渡しレイヤー | OSレイヤーで動作する汎用・オープンなミドルウェア |

# **3\. システム設計仕様**

## **3.1 全体アーキテクチャ**

CR-Bridgeは4つの独立したサブシステムから構成され、それぞれがRustの非同期ランタイム（tokio）上で並行動作する。サブシステム間はゼロコピー共有メモリ（後述のSpatial Ledger）を介して通信し、プロセス間コピーを発生させない。

| **レイヤー** | **コンポーネント / 役割** |
| --- | --- |
| プロトコル層 | Anti-Teleport Protocol (ATP)：予測型・非等時性同期エンジン |
| 計算層 | Spatial Math Accelerator (SMA)：SIMD/FMA/NPU空間演算ライブラリ |
| データ構造層 | Shared Memory Spatial Ledger (SMSL)：ゼロコピー空間DB |
| ブリッジ層 | Protocol Bridges：VRChat/SNS/ARグラスへのアダプター群 |

## **3.2 プロトコル層：Anti-Teleport Protocol (ATP)**

### **3.2.1 問題の定式化**

座標 x_t を持つエンティティが、パケットロス率 p のネットワークを通じて受信端に届く場合、素朴な実装では受信されたパケット x_{t+k} が直接適用され、k フレーム分の瞬間移動が観測される。ATPはこれを以下の手順で解決する。

### **3.2.2 パケット構造（FlatBuffers スキーマ）**

| **フィールド** | **型** | **説明** |
| --- | --- | --- |
| entity_id | uint64 | エンティティの一意識別子 |
| timestamp_us | uint64 | 送信側マイクロ秒タイムスタンプ |
| position | Vec3f | 現在座標 \[x, y, z\] |
| velocity | Vec3f | 速度ベクトル \[vx, vy, vz\] m/s |
| acceleration | Vec3f | 加速度 \[ax, ay, az\] m/s² |
| orientation | Quaternionf | 姿勢クォータニオン \[qw, qx, qy, qz\] |
| angular_velocity | Vec3f | 角速度 \[wx, wy, wz\] rad/s（IMUデータ） |
| imu_confidence | float32 | IMUデータ信頼度 \[0.0-1.0\] |

通信プロトコルにはHTTP/3（QUIC）を採用し、エンティティごとに独立したストリームを割り当てる。これによりHead-of-Line Blockingを排除し、1エンティティのパケットロスが他エンティティの受信に影響しない設計とする。

### **3.2.3 予測補間パイプライン**

- デッドレコニング：速度・加速度・IMU角速度を使い、パケット欠落中の位置・姿勢を各フレームで再計算する。
- 拡張カルマンフィルタ (EKF)：状態ベクトル \[x, v, a, q, ω\] の事前推定と、次パケット到着時の更新を繰り返す。非線形な回転成分（クォータニオン）はSO(3)多様体上の一次近似で扱う。
- 動的ジッターバッファ：ネットワーク遅延分散 σ_t をEWMAで追跡し、バッファ量を \[σ_t × 1.5\] msに自動調整。パケット欠落時は「慣性予測位置」を使用し、再到着時にエルミート補間（C1連続）で実座標へ滑らかに収束させる。
- CPU実装：FMA (Fused Multiply-Add) 命令を活用し、カルマンゲイン行列演算を数値誤差なく高速処理する。1エンティティあたりの計算コスト目標は < 5μs / update。

## **3.3 計算層：Spatial Math Accelerator (SMA)**

### **3.3.1 SIMD並列座標変換**

VRChatのワールドには数十〜数百のアバターが同時に存在し、各アバターの座標をARグラスの視点からのビュー行列に変換する計算が毎フレーム必要となる。SMAはこの計算をSIMD命令で並列化する。

- AVX-512 (x86-64)：512ビットレジスタを使い、1命令で16個のfloat32値（= 5つのVec3 + α）を処理する。アバター100体のビュー変換を、スカラー実装比で約16倍のスループットで処理する。
- NEON (ARM/Android)：128ビット×4レーン。Androidスマートフォン・ARグラス向け実装。
- Rustのstd::arch::x86_64 / std::arch::aarch64 intrinsicsで直接記述し、コンパイラの自動ベクトル化に依存しない。
- キャッシュ効率：AoS（Array of Structures）ではなくSoA（Structure of Arrays）レイアウトでエンティティデータを格納し、SIMD処理時のストライドアクセスを排除する。

### **3.3.2 NPUによる視線予測プリフェッチ**

最近のモバイルSoC（Qualcomm Snapdragon 8シリーズ、Apple M3等）に搭載されているNPUを利用し、軽量LSTM/GRUモデルを推論する。

- 入力：直近200msの頭部角速度・視線ベクトル・IMUデータ。
- 出力：50ms先の視線方向確率分布（視錐台の事前推定）。
- 用途：次フレームの視錐台を事前推定し、その範囲内のエンティティデータをSMSL（Spatial Ledger）からプリフェッチすることで、描画エンジンへのデータ提供レイテンシをゼロに近づける。
- モデルサイズ：<500KB（NPU上で推論 < 2ms）。フレームレートの予算を消費しない設計とする。

## **3.4 データ構造層：Shared Memory Spatial Ledger (SMSL)**

### **3.4.1 ゼロコピーアーキテクチャ**

SMSLは、ネットワーク受信からAR描画エンジンまで、データが一切コピーされない経路を実現するインメモリ空間データベースである。

- mmap共有メモリ領域：全プロセス（ATPデーモン・SMAライブラリ・ARレンダラー）が同一の物理メモリページを参照する。ページのオーナーシップはSMSLが管理し、参照カウントで解放タイミングを制御する。
- ゼロコピーデシリアライズ：受信したFlatBuffersバイナリを構造体にキャストして直接参照する。パースオーバーヘッドは実質ゼロ。
- ロックフリー設計：SeqLock（シーケンスロック）を用いた単一ライター複数リーダーの構造。SIMDコアから書き込み、描画コアから読み出す際にmutexを使わない。

### **3.4.2 空間インデックス**

- S2 Geometry（Googleのセル分割ライブラリ）をベースに、現実世界座標と仮想空間座標を統一された階層インデックスで管理する。
- 視錐台クエリ：ARグラスの現在視錐台（6面体）とNPU予測視錐台を共にインデックスに照合し、その内部に存在するエンティティのみをプッシュ配信する。O(log N)クエリ計算量。
- NUMA最適化：受信処理コア・計算処理コア・描画コアをそれぞれ別の物理コアに固定し、各コアのローカルNUMAノードにデータを配置してバス競合を回避する。

## **3.5 ブリッジ層：Protocol Bridges**

### **3.5.1 VRChat OSC Bridge**

VRChatは公式にOSC（Open Sound Control）プロトコルで内部変数を外部公開している。本ブリッジはこれを受け取りATP形式に変換する。

- 入力：OSC /avatar/parameters/\* （ボーン位置・姿勢・表情等）
- 処理：受信したOSCパケットを FlatBuffers フォーマットに変換後、SMSLに書き込む。ATPの予測エンジンがこれを読み取り、AR側に補間済みデータを流す。
- OSCはUDP上で動作するためパケットロスが起き得るが、ATPの予測補間がこれを吸収する。

### **3.5.2 SNS Spatial Bridge**

既存SNSは位置情報を持つコンテンツを提供するが、それを空間的にクエリするAPIを持たない。本ブリッジが「空間の概念」を後付けする。

- Twitter/X：Streaming API（Filtered Stream）を使い、地名・座標を含むツイートをリアルタイム取得。地名はジオコーダー（Nominatim/Google Geocoding）で緯度経度に変換し、SMSLのS2インデックスに登録する。
- Instagram：Graph APIのMedia with Locationエンドポイントを使い、位置情報付きメディアを取得してSMSLに登録する。
- 各SNSの制限（レートリミット等）に応じてキャッシュ有効期限をエンティティごとに設定し、鮮度を管理する。

### **3.5.3 AR-VPS Bridge**

Visual Positioning System（視覚的位置推定）により、スマートフォン/ARグラスのカメラ映像から世界座標を確定する。

- スマートフォン側：カメラフレームから特徴点（ORB等）を抽出し、軽量記述子をサーバーへ送信する。画像全体を送らないため通信量を大幅に削減する。
- サーバー側：事前構築した3Dポイントクラウドマップと照合（Visual Bag-of-Words）し、6DoFカメラ姿勢を返す。
- 本提案期間内では東京都内の特定エリアをパイロットマップとして構築し、実証実験を行う。（ハードウェアコスト：マップ構築用カメラ機材〜¥50,000）

# **7\. 新規性・独自性**

本提案の最大の独自性は「既存SNSの投稿データに空間座標を後付けし、ARの視錐台ベースのリアルタイム配信に組み込む」という設計にある。Nianticはこれに最も近い取り組みをしているが、クローズドエコシステムであり外部開発者が低レイヤーAPIにアクセスする手段がない。

CR-BridgeはOSS・標準準拠（OpenXR・QUIC・FlatBuffers）で設計され、特定ハードウェアや特定クラウドに依存しない「中立なインフラ」として機能する。これはMetaやAppleが主導するクローズドなXR基盤に対する、オープンな対抗軸となりうる。

また、ATPの予測補間アルゴリズムとSMSLのゼロコピー設計の組み合わせは、著者が文献調査した限り同等の実装が公開されておらず、本提案による実装と性能評価が当該分野の技術的知見に貢献できると考える。
